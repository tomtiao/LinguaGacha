# 项目文档

这份 skill 是长期文档过滤器，不是文档维护说明书。默认假设文档已经过载；目标是把稳定规则收口到唯一权威源，并通过删除、合并、迁移、压缩降低维护负担。

## 目标形态

长期项目文档只收口到：

```text
AGENTS.md
docs/
  ARCHITECTURE.md
  API.md
  FRONTEND.md
  WORKFLOW.md
  DATA.md
```

`PRODUCT.md` 是 `$impeccable` 的产品上下文，`DESIGN.md` 是设计权威；二者不是 `docs/` 下的并行总纲。生成物、归档、任务临时文档、外部规范镜像、用户明确保留或稍后手动处理的计划文件，不自动吸收为长期权威文档。

## 收录闸门

一条信息必须同时满足四项才进入长期文档：

| 问题 | 不满足时 |
| --- | --- |
| 不知道它会改错落点、边界、契约或验证吗？ | 删除 |
| 它无法从代码、目录、类型名或局部实现直接看出吗？ | 删除 |
| 它描述当前稳定事实，而不是历史过程或本次改动吗？ | 删除或改写 |
| 它有且只有一个固定归宿吗？ | 合并、迁移或删除 |

新增内容必须替换、合并、迁移、压缩旧信息，或补上会导致维护误判的真实缺口。只是“更完整”“更保险”“顺便提醒”的内容不写。

## 唯一归宿

| 信息回答的问题 | 唯一归宿 |
| --- | --- |
| Agent 如何协作、编码和交付 | `AGENTS.md` |
| 系统如何分层、跨层边界和阅读路径在哪里 | `docs/ARCHITECTURE.md` |
| HTTP / SSE / bootstrap / topic / 错误码 / mutation 契约是什么 | `docs/API.md` |
| Electron / preload / renderer / `ProjectStore` 消费边界是什么 | `docs/FRONTEND.md` |
| 任务起手式、验证矩阵、文档同步与交付自检是什么 | `docs/WORKFLOW.md` |
| Python Core 数据域、状态拥有者、唯一写入口、SQL 落点是什么 | `docs/DATA.md` |
| 产品语义、设计语言、视觉权威来源是什么 | `PRODUCT.md` 或 `DESIGN.md` |

如果一段内容同时属于多个归宿，拆开；如果同一规则出现在多个归宿，只保留权威版本，其他位置改成引用或删除。

## 执行顺序

动笔前先列出本轮信息集合决策：删什么、并什么、移什么、压缩什么、补什么、保留什么。随后按下面顺序执行：

1. 删除不满足收录闸门的信息。
2. 合并重复规则和并行权威。
3. 迁移错位内容到唯一归宿。
4. 压缩目录复述、低密度表格和历史叙事。
5. 只有证据充分时才补写缺口。

补写证据只能来自当前代码、现有权威文档或用户明确确认。证据不足时列为未决，不写成长期规则。

结构性文档任务如果没有发生删除、合并、迁移或压缩，默认没有完成；除非能说明现有文档已经干净，只剩必要补写。

## 必须重组

出现下面任一信号，重写相关信息块，不要局部补一句：

- 同一主题在一份文档或多份文档中重复出现。
- 新规则只能靠“补充、另外、注意、同时”塞进去。
- 旧章节名、旧目录划分或历史叙事仍在解释当前规则。
- 表格和列表继续变长，但没有改变维护判断。
- `AGENTS.md` 承担专题正文，或专题文档反向承载协作规则。

## 删除前检查

删除遗留文档、并行文档或入口文件前，必须确认：

- 稳定规则已经迁移到唯一归宿。
- 仓库内链接、脚本、README、测试断言和技能提示不再指向旧入口。
- 它不是工具链入口、外部流程入口、用户指定保留文件或生命周期未明的证据材料。

## 交付

交付时只汇报信息集合变化：

- 删除：删掉了哪些不合格信息或旧入口。
- 合并：哪些重复规则被收成唯一权威。
- 迁移：哪些信息移动到哪个归宿。
- 压缩：哪些长段、表格或历史叙事被压成判断规则。
- 补写：补了什么缺口，证据是什么。
- 保留：看似可疑但仍保留的理由。
- 未决：证据不足而没有写入长期文档的问题。
- 验证：链接、引用、固定文档形态和 diff 自检结果。

若交付主要是新增内容，必须说明它替换、合并、迁移或压缩了什么；否则视为只增不减。
